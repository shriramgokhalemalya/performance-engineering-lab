# Performance Engineering Lab - Login Service Context

Updated: 2026-07-14
Repository: https://github.com/shriramgokhalemalya/performance-engineering-lab
Current branch: main

## 1) Project Purpose

This service is a Spring Boot microservice used as a performance engineering playground, not a production auth system.

Primary learning areas:
- Spring Boot 3.5
- Spring Security with stateless JWT auth
- Oracle JDBC integration
- Docker multi-stage builds
- Kubernetes (Kind and Docker Desktop)
- Prometheus/Grafana monitoring
- Performance diagnostics (memory pressure, GC logs, heap dumps)

## 2) Tech Stack

- Java: 17

## 8) Security Model

From `SecurityConfig`, `JwtService`, `JwtAuthenticationFilter`:
- Stateless session policy
- CSRF disabled
- HTTP Basic disabled
- Form login disabled
- JWT filter executes before `UsernamePasswordAuthenticationFilter`
- Missing/invalid token on protected endpoints returns 401
- `UserDetailsService` uses empty `InMemoryUserDetailsManager` to avoid generated default credentials
- JWT token subject = username, signed with configured secret

Current hardcoded login in `AuthService`:
- username: `admin`
- password: `password`

## 9) User Cache Behavior (Current)

From `UserService`:
- Uses a single in-memory cached list (`cachedUsers`)
- `findAllUsers()` is synchronized
- First read -> cache miss -> load from DB -> cache set
- Subsequent reads -> cache hit -> same cached instance reused
- `addUser()` is synchronized
- On successful insert, cache is refreshed from DB (write-through refresh)

Metrics emitted:
- `login_service_users_cache_hit_total`
- `login_service_users_cache_miss_total`
- `login_service_users_cache_refresh_total`
- `login_service_users_cache_size`
- `login_service_users_cache_size_bytes`

## 11) Docker

### Dockerfile
- Multi-stage build:
  - Build stage: `maven:3.9-eclipse-temurin-17`
  - Runtime stage: `eclipse-temurin:17-jre`
- Copies built JAR to `/app/app.jar`
- Exposes `8080`
- Entry point: `java -jar app.jar`

### Dockerfile.jdk
- Same multi-stage build pattern
- Runtime uses `eclipse-temurin:17-jdk`
- Sets `SERVER_PORT=8081`
- Exposes `8081`
- Enables JVM tuning and diagnostics:
  - `-Xms512m -Xmx1024m`
  - Heap dump on OOM
  - GC logging to `/app/gc.log`

Common local run command used:

```powershell
docker run -d --name login-service -p 8081:8081 login-service:latest
```

Health check:

```text
http://localhost:8081/actuator/health
```

## 16) Important Caveats to Reconcile

1. Port alignment needs review across runtime targets.
- App default port in properties is 8081.
- `Dockerfile` runtime exposes 8080.
- Kubernetes deployment/service target 8080.

2. Ingress manifest structure should be revalidated.
- `k8s/ingress.yaml` currently appears to have `host` and `http` split across separate list items, which is likely not the intended schema shape.

3. Security debug logging is enabled globally.
- Useful for development, but noisy for high-load tests and can affect performance.
